# Utility Tools Design

**Date:** 2026-04-25
**Status:** Approved

## Overview

Add a small, fixed catalogue of utility tools that gem admins can enable per gem. When enabled, tools are injected into the Pydantic AI agent at run time, following the same pattern as document tools. The initial catalogue has two tools: **datetime** (current date/time) and **calculator** (mathparse expression evaluator).

---

## Backend

### `backend/tasks/models.py`

Add `ToolId` enum alongside the existing model types:

```python
class ToolId(str, Enum):
    DATETIME = "datetime"
    CALCULATOR = "calculator"
```

Add `tools` field to `Task`:

```python
class Task(BaseModel, frozen=True):
    # ... existing fields ...
    tools: frozenset[ToolId] = frozenset()
```

`ToolId` lives in `models.py` (not `tools.py`) to avoid a circular import: `Task` references it, and `tools.py` imports from `models.py`.

### `backend/tasks/tools.py` (new)

Owns all tool logic. Exports:

- `build_tool_functions(tool_ids: frozenset[ToolId]) -> list[Callable]` — returns the callable(s) for the requested tool IDs in a stable order (ToolId enum order).

Tool function implementations (these are the callables passed to the agent):

- `get_current_datetime() -> str` — returns `datetime.now()` formatted as a human-readable string.
- `calculate(expression: str) -> str` — passes the expression to `mathparse.parse()`; wraps exceptions and returns an error string rather than raising, so a bad expression doesn't crash the agent.

Both functions carry precise docstrings and type hints. Pydantic AI reads these to generate the tool description for the model — no separate system prompt management is needed. A module-level `_TOOL_REGISTRY: dict[ToolId, Callable]` maps each ID to its callable.

### `backend/tasks/runner.py`

`_make_agent()` gains a `tool_ids: frozenset[ToolId] | None = None` parameter. It combines document tools and utility tools into a single list passed to `Agent(model, tools=tools, ...)`. Pydantic AI derives tool descriptions from each callable's docstring and type hints automatically — no manual system prompt construction for utility tools. The existing `_DOC_SYSTEM_PROMPT` is kept for document tools (it explains the document library concept, which is not self-evident from the function signatures alone). No tools → `Agent(model)` unchanged.

`stream_task` and `run_task` both pass `task.tools` to `_make_agent()` — they already receive the full `Task` object, so no signature changes are needed.

### `backend/tasks/repository.py`

`tools` column added directly to `_CREATE_TABLE` DDL (no migration; project is in development). Stored as a JSON array of string values (e.g. `["datetime"]`). `_row_to_user_task` reconstructs `frozenset[ToolId]`; `save_task` serialises it.

### `pyproject.toml`

`mathparse` added to production dependencies.

## Frontend

### `frontend/src/api/tasks.ts`

```typescript
export type ToolId = 'datetime' | 'calculator'
export const TOOL_IDS: { id: ToolId; labelKey: string; descKey: string }[] = [
  { id: 'datetime',   labelKey: 'gem.tools.datetime',   descKey: 'gem.tools.datetime.description' },
  { id: 'calculator', labelKey: 'gem.tools.calculator', descKey: 'gem.tools.calculator.description' },
]
```

`UserTask` interface gains `tools: ToolId[]`.

### `frontend/src/pages/GemForm.tsx`

`blankGem()` initialises `tools: []`.

A new **"Tools"** section is added below the "Capabilities" section. It contains:

1. A `HelpBar` with `help.gem.tools.{prompt,title,content}` keys.
2. A short hint line.
3. One `Checkbox` + `Label` pair per tool in `TOOL_IDS`, with a one-line description under the label. Toggling a checkbox adds/removes the tool ID from `gem.tools`.

The tool list is static (hardcoded in `TOOL_IDS`); no API endpoint is needed to enumerate tools.

## Locale

New keys added to `locales/en.json` (and translated to all other locales):

| Key | Value |
|-----|-------|
| `gem.tools.section_title` | `Tools` |
| `gem.tools.section_hint` | `Allow the model to call these built-in tools during a run.` |
| `gem.tools.datetime` | `Date & time` |
| `gem.tools.datetime.description` | `Lets the model look up the current date and time.` |
| `gem.tools.calculator` | `Calculator` |
| `gem.tools.calculator.description` | `Lets the model evaluate a mathematical expression.` |
| `help.gem.tools.prompt` | `What are tools?` |
| `help.gem.tools.title` | `Tools` |
| `help.gem.tools.content` | `Tools let the model perform actions during a run — like checking the current date or doing a calculation — before writing its answer. Only enable tools that are relevant to what this gem does.` |

Translations are generated by a Haiku agent at implementation time. All locale files (`de`, `es`, `fr`, `it`, `zh`) receive the same keys.

---

## Docs

The admin guide's gem creation page (`docs/en/admin/`) gets a new "Tools" section explaining:

- What tools are and when to enable them.
- The two available tools (datetime, calculator) with a sentence each.
- A note that tools consume context window space on every run, so only enable what is needed.

---

## Out of scope

- A `/api/tools` listing endpoint (static frontend list is sufficient).
- Trigonometric functions (mathparse does not support them; may be revisited if mathparse is forked/extended).
- Code interpreter tools (injection risk).
- Per-tool usage reporting or audit logging.
